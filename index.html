MySQL慢查询优化与Explain

发表于 2021-09-27 更新于 2021-09-29 分类于 数据库 ， MySQL
概念 解释
慢查询物理原因 出现大量磁盘随机IO时，大部分时间都被浪费在寻址而非传输数据。
解决方案为建立索引，减少随机IO。
慢查询索引原因 1、索引分区度低。
2、索引创建太多。
3、常用字段/排序字段没有创建索引。
慢查询SQL原因 1、对索引进行运算、使用函数、模糊匹配、隐式转换等。
2、联合索引不包含前面的索引，根据最左匹配原则会导致索引失效。
SQL优化 1、使用join代替子查询。
2、使用覆盖索引，减少回表。
3、where将过滤数据多的条件放在前面。
4、使用联合索引，而不是建立多个索引。
Explain排查 使用Explain关键字查看SQL语句的执行信息。
type、key、extra字段观察是否使用了索引，是否进行的是全表扫描。

Hive SQL

发表于 2021-09-25 分类于 Hive
概念 解释
HiveSQL转MR（普通） 普通的任务直接用fetch的方式直接读取数据，不需要转MR。
HiveSQL转MR（Join） Map：生成键值对，以JOIN ON条件中的列作为Key，以JOIN之后所关心的列作为Value，在Value中还会包含表的 Tag 信息，用于标明此Value对应于哪个表。
Shuffle：将键值对发送至不同的Reducer中。
Reduce：Reducer通过 Tag 来识别不同的表中的数据，根据Key值进行Join操作。
HiveSQL转MR（Group By） Map：生成键值对，以GROUP BY条件中的列作为Key，以聚集函数涉及的字段作为Value。
Shuffle：将键值对发送至不同的Reducer中。
Reduce：根据SELECT子句的列以及聚集函数进行Reduce。
HiveSQL转MR（Distinct） 相当于没有聚集函数的GROUP BY，操作相同，只是键值对中的value可为空。
Hive SQL与Spark SQL 1、Hive SQL底层是MR，Spark SQL底层是RDD的计算。本质上两者的区别变成了MR和Spark的区别（内存、算子、容错）。

2、对于Hive本身来说，它包括metastore（元数据），就是schema的管理，比如有哪些表，哪些字段，什么数据类型，有没有索引等。而Spark只是一个计算引擎，没有元数据管理工具。

3、hive有hive on mapreduce和hive on spark。hive on spark 和Spark SQL并没有特别明显的性能优化（毕竟计算引擎都是Spark）。

4、hive on mapreduce：适合处理离线非实时数据，基于磁盘的处理使得性能上较为稳定。
Spark SQL：实时性要求或者速度要求较高的场所，基于内存的处理可能会出现数据倾斜造成反压等问题，导致内存溢出等错误。

在Driver端生成可执行代码和DAG有向无环图。
Execution
执行器		 执行代码。 计算结果。

0924 CDG一面

发表于 2021-09-24 更新于 2021-09-26 分类于 面试

海量数据找Top 100账号出现次数

1、Hash(账号) % 桶数量 = 小文件id。把账号分离到多个小文件中。

理想情况下每个小文件的大小应该恰好等于内存容量

2、对于每一个小文件，在内存中构建一个账号为Key，统计次数为Value的一个HashMap，统计每个账号的出现次数。

把HashMap的结果保存在磁盘上。

如果小文件数量远大于100个，那么对每个小文件就只需要保存统计次数最多的那个账号。

3、在内存中构建一个有100个节点的最小堆，堆中的每个节点保存了HashMap中的Key-Value的数据，只是堆调整的依据是Value值，也就是统计次数。

将每个小文件对应的HashMap读取进入内存，把Key-Value数据入堆，最后剩下的账号就是Top 100。

如果内存足够，也可以直接把账号-次数的数据读取进来做全局排序，因为此时每个小文件只有一个账号。

如果每个HashMap太大，那么就需要在第2步构建一个按照Value排序的HashMap，然后取前100个数据保存下来即可。

Spark调优

发表于 2021-09-24 分类于 Spark

开发调优

概念 解释
1、避免创建重复的RDD 避免进行多次重复计算来创建同一个RDD增加计算开销。
2、尽可能复用同一个RDD 例如有一个RDD的数据格式是key-value类型的。
另一个是单value类型的。
这两个RDD的value数据是完全一样的。
那么此时我们可以只使用key-value类型的那个RDD，
因为其中已经包含了另一个的数据。
3、对多次使用的RDD进行持久化 重复出现的RDD会被重复计算
rdd.persist(StorageLevel.MEMORY_AND_DISK_SER)
4、尽量避免使用shuffle类算子 尽可能避免使用reduceByKey、join、distinct、repartition等会进行shuffle的算子。
尽量使用map类的非shuffle算子。
如将join转换成Broadcast+map。
5、使用map-side预聚合的shuffle算子 reduceByKey和aggregateByKey算子都会对每个节点本地的相同key进行预聚合。
groupByKey算子是不会进行预聚合的，全量的数据会在集群的各个节点之间分发和传输。
6、使用高性能的算子 使用mapPartitions替代普通map。
使用foreachPartitions替代foreach。
使用filter之后进行coalesce操作。
使用repartitionAndSortWithinPartitions替代repartition与sort类操作。
7、广播大变量 这样每个Executor只有一个副本，而不是每个Executor中的每个Task都有一个副本。
8、使用Kryo优化序列化性能 Kryo序列化类库的性能比Java序列化类库的性能要高很多。
9、优化数据结构 使用字符串替代对象，使用原始类型（比如Int、Long）替代字符串，使用数组替代集合类型。
尽可能地减少内存占用，降低GC频率，提升性能。

数据倾斜调优

概念 解释
数据倾斜发生的现象 1、绝大多数task执行得都非常快，但个别task执行极慢。
2、原本能够正常执行的Spark作业，某天突然报出内存溢出异常。（数据太大资源不足）
数据倾斜排查 1、yarn-client模式提交，本地直接看log，在log中找到当前运行到了第几个stage。
2、yarn-cluster模式提交，通过Spark Web UI来查看当前运行到了第几个stage。
3、两种模式下都可以在Spark Web UI上查看当前stage各个task分配的数据量，从而进一步确定是不是task分配的数据不均匀导致了数据倾斜。
4、确定数据倾斜发生在哪个stage后，根据stage划分原理，推算出发生倾斜的那个stage对应代码中的哪一部分，这部分代码中肯定会有一个shuffle类算子。
查看Key的分布情况 1、Spark SQL：group by、join语句导致数据倾斜，直接从源表中查看。
2、Spark RDD：shuffle算子，加入查看key分布的代码，比如RDD.countByKey()，然后用collect/take到客户端打印一下。
数据倾斜解决方案 1、使用Hive ETL预处理数据。
2、过滤少数导致倾斜的key。（null值）
3、提高shuffle操作的并行度。
4、两阶段聚合（随机前缀局部聚合+全局聚合），仅适用于reduceByKey等聚合操作，不适用于join连接操作。
5、将reduce join转为map join。（大表join小表的场景）
6、采样找出有数据倾斜的key，拆出来，有数据倾斜的表加随机前缀，无倾斜的表做数据膨胀保证能被join上（大表join大表的场景，表1有数据倾斜，表2没有）
