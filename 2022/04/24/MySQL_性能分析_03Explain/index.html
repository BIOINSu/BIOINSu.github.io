MySQL 性能分析：Explain

发表于 2022-04-24 更新于 2022-04-26 分类于 数据库 ， MySQL

简介

如果想查看某个查询的执行计划的话，可以在具体的查询语句前边加一个EXPLAIN ：

EXPLAIN 语句输出的各个列的作用如下：

注意，EXPLAIN并不会真正去执行SQL语句，而只是返回根据统计信息得到的执行计划。

下文案例中的表结构为：

table

table字段表示当前语句需要查询的表的名称。

不论查询语句有多复杂，里面包含了多少个表，到最后也是需要对每个表进行单表访问的，所以MySQL规定EXPLAIN语句输出的每条记录都对应着某个单表的访问，该条记录的table列代表着该表的表名（有时不是真实的表名字，可能是简称）。

id

id字段表示每个查询。每个SELECT关键字都对应一个唯一的id。

  • id如果相同，可以认为是一组，执行顺序从上往下顺序执行。

  • 在所有组中，id值越大，优先级越高，越先执行。（特别是子查询中出现）

  • 关注点：每个id，表示一趟独立的查询, 一个sql的查询趟数越少越好，即id号越小越好。

单表查询：

双表关联：

id相同，查询顺序从上往下顺序执行，不是嵌套的关系。

嵌套子查询：

子查询的id更大，更优先被执行，查询之间是嵌套执行的关系。

重写的嵌套子查询：

即使是子查询，id相同，说明并非嵌套的关系，查询语句被重写为顺序执行的多表关联形式。

UNION存在临时表：

没有id的表，说明是临时表。这里是UNION操作是需要去重的，因此需要构建临时表进行去重操作。

UNION ALL不存在临时表：

不需要去重操作，因此不需要创建临时表。

select_type

SIMPLE：不包括UNION或者子查询的简单查询

PRIMARY、UNION、UNION RESULT：子查询以及包含UNION或UNION ALL的情况

最左边查询的select_type为PRIMARY，其余的为UNION。

子查询中最外层的查询也为PRIMARY。

使用临时表进行去重时，产生临时表的查询对应的select_type为UNION RESULT。（UNION ALL就不会产生临时表）

SUBQUERY：不相关子查询

包含子查询的语句不能转换为“semi-join“的形式（即表关联的形式），并且该子查询不是相关子查询（即子查询用到外查询的表作为过滤条件），则子查询的select_type为SUBQUERY。

DEPENDENT SUBQUERY：相关子查询

包含子查询的语句不能转换为“semi-join“的形式（即表关联的形式），并且该子查询是相关子查询（即子查询用到外查询的表作为过滤条件），则子查询的select_type为DEPENDENT SUBQUERY。

这样的子查询可能被执行多次，因为外层每次朝内层传递一个值，内层子查询就会执行一次。

DEPENDENT UNION：含有UNION的子查询

在包含UNION或UNION ALL的子查询时，如果各个子查询都依赖于外层查询的话，最左边的小查询select_type为DEPENDENT SUBQUERY，其余的就是DEPENDENT UNION。

下面的例子本质上是因为将IN语句改写为了EXSISTS语句，因此会重构为依赖于外层的子查询，所以含有DEPENDENT UNION。

DERIVED：含有派生表的查询

包含派生表（由查询语句生成的表）的查询语句，生成该派生表的查询语句，对应的select_type为DERIVED。

MATERIALIZED：物化表

子查询的结果被转化为物化表，供外层的结果进行表关联时，该子查询对应的select_type为MATERIALIZED

type

所有访问方法从优到劣可以排序为： system ， const ， eq_ref ， ref ， fulltext ， ref_or_null ，index_merge ， unique_subquery ， index_subquery ， range ， index ， ALL 。

从访问方法可以大约地看出查询语句的执行效率。

结果值从最好到最坏依次是： system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > ALL。其中比较重要的几个由粗体标识。

SQL 性能优化的目标：至少要达到 range 级别，要求是 ref 级别，最好是 consts级别。（阿里巴巴开发手册要求）

system

当表中只有一条记录，并且该表使用的存储引擎中统计数据时精确的，那么对表的访问方法是system。

这里存储引擎实际上就是MyISAM或Memory，因为他们额外存储了一些统计数据比如表中的数据条数，使用时直接获得不需要去表中查，而InnoDB则需要查询表中的数据才能获得精确的统计信息。

const

根据主键或者唯一性约束的字段，与常数进行等值匹配，那么对表的访问方法是const。

eq_ref

在表关联时，若被驱动表通过主键或唯一性约束的字段，和驱动表进行等值访问时，对被驱动表的访问方法是eq_ref。

如果该主键或唯一性约束的字段是联合索引，所有联合索引中的字段都必须进行等值比较。

从执行计划的结果中可以看出，MySQL打算将s2作为驱动表，s1作为被驱动表，重点关注s1的访问方法是eq_ref ，表明在访问s1表的时候可以通过主键的等值匹配来进行访问。

也就是每次将s2中字段id的值取出来在s1中查询。

ref以及ref_or_null

使用普通的二级索引（即非聚簇索引）进行等值匹配时，那么对表的访问方法是ref。

当该字段上的值可以是NULL时，对表的访问方法是ref_or_null。

index_merge

查询条件中的字段存在索引，并且使用OR连接时，那么对表的访问方法是index_merge，即采用索引合并的方式进行查询。

从执行计划的type 列的值是index_merge 就可以看出，MySQL 打算使用索引合并的方式来执行对s1 表的查询，即两个索引都使用到了。

unique_subquery

是针对子查询中表的访问方法。

当一个查询语句中有包含“IN”的子查询；如果查询优化器决定将“IN”子查询转换为“EXISTS”子查询；且子查询可以用到主键进行匹配，那么对表的访问方法是unique_subquery。

可以看出，外层是ALL，因为每次都需要从外层取出一条数据放入子查询，子查询中其实变成了将是s1.key2作为表s2的主键id进行查询，且满足条件s1.key1 = s2.key1，因为查询时用到了表s2的主键，因此对表s2的访问方法是unique_subquery。

index_subquery

和上面的情况相同，不同点是子查询使用的是普通的二级索引进行匹配，那么对表的访问方法是index_subquery。

range

如果使用索引获取某个范围内的记录，则对表的访问方法是range。

index

当发生了索引覆盖（要查询的字段刚好是索引所在的字段，不需要根据主键回表），但是需要扫描索引中的全部记录时（联合索引中不满足最左匹配原则时就无法使用索引，必须在这个联合索引上进行全部扫描），则对表的访问方法是index。

all

全表扫描

possible_keys和key

key并非是possible_keys的子集，possible_keys只要是筛选字段中存在的索引都会列出。但是假如SQL中未写筛选条件，而优化器优化后能够使用索引，则possible_keys是NULL，但key不是NULL。

key_len

使用的索引字段类型所占的字节数，对于联合索引比较有意义，在联合索引下，占用字节数越多说明使用到了更多的联合索引中的索引。

varchar类型，utf-8字符集下一个字符占3个字节，再加上可能为空的1个字节，和记录总的长度信息的2个字节，一共是303个字节。

使用了联合索引中的多个字段。

ref

当使用索引列进行等值查询时，ref字段表示进行等值匹配时的对象信息。比如是一个常数，或几个列。

表示匹配的对象经过了函数处理。

rows

预估需要读取的记录条数，并非最终查询的结果记录条数。

通常来说，rows需要越小越好。

个人理解：在MySQL中仅仅根据给出的关于这个索引的条件和索引本身，来判断需要扫描多少行。

MYSQL 在真正开始执行语句之前，并不能精确地知道满足这个条件的记录有多少条，而只能根据统计信息来估计记录数。

这个统计信息就是索引的“区分度”。显然，一个索引上不同的值越多，这个索引的区分度就越好。而一个索引上不同的值的个数，我们称之为“基数”（cardinality）。也就是说，这个基数越大，索引的区分度越好。

采样统计的时候，InnoDB 默认会选择 N 个数据页，统计这些页面上的不同值，得到一个平均值，然后乘以这个索引的页面数，就得到了这个索引的基数。

链接：https://juejin.cn/post/6864517310649286669

filtered

某个表经过搜索条件过滤后，剩余记录条数的百分比。和rows配合起来看，rows x filtered% = 大约需要查询的数据条数。

表明内层子查询大概需要执行9688 x 10% = 968次，因为这是外层查询传入内层查询的数据条数。

Extra

没有过滤条件

Impossible WHERE

过滤条件恒为false，此时无法检索出任何数据。

Using where

使用全表扫描（无索引），并且WHERE中有针对该表的搜索条件，在EXTAR字段中显示Using where。

使用了索引，且WHERE中除了索引列外还有其他字段，在EXTAR字段中显示Using where。（如果WHERE中只有索引列，则不提示任何东西）

No matching min/max row

存在MIN/MAX的聚合函数却过滤不出任何数据时。

Using index

存在覆盖索引的情况时（索引中包括了需要查询的字段，不需要根据主键回表时）显示。

Using index condition

使用了索引条件下推的情况，即在索引中就能够过滤出数据再进行回表，而不是先回表再过滤数据。

在索引的B+树中就将满足后缀的数据筛选出来再进行回表，而不是先回表再筛选满足后缀的数据。

Using join buffer (Block Nested Loop)

执行表关联时，被驱动表不能利用索引有效加快速度，MySQL分配一块称为“join buffer”的内存空间加速查询，即基于块的嵌套循环算法。

Not exists

左外连接，且WHERE条件中被驱动表的过滤条件要求某个列等于NULL，但是那个列本身不允许为NULL时，显示Not exists。

Using intersect(...) 、Using union(...) 和Using sort_union(...)

Using filesort

如果查询语句中存在ORDER BY 索引字段，则可以通过索引直接获得排序的结果。

Using temporary

partitions

其他

  • EXPLAIN不考虑各种Cache，即没有进行缓存优化

  • EXPLAIN不能显示MySQL在执行查询时所作的优化工作，只能显示结果

  • EXPLAIN不会告诉你关于触发器、存储过程的信息或用户自定义函数对查询的影响情况

  • 部分统计信息是估算的，并非精确值（rows）

Explain的四种输出方法

传统格式

即最普通的表格。

JSON格式

传统格式中缺乏了衡量执行计划好坏的重要属性：成本。

在EXPLAIN单词和真正的查询语句中间加上FORMAT=JSON即可以JSON格式展示更为详细的结果：

其中找到"cost_info"字段里的详细信息，即描述了成本：

  • read_cost 是由下边这两部分组成的：

    • IO 成本。
    • 检测rows × (1 - filter) 条记录的CPU 成本。（即过滤数据的成本）

    rows和filter都是前边介绍执行计划的输出列，在JSON格式的执行计划中，rows相当于rows_examined_per_scan，filtered名称不变。

  • eval_cost ：检测 rows × filter 条记录的成本。（即保留数据的成本）

  • prefix_cost 就是单独查询表的成本，也就是：read_cost + eval_cost（总的查询成本）

  • data_read_per_join 表示在此次查询中需要读取的数据量。

由于s2 表是被驱动表，所以可能被读取多次，这里的read_cost 和eval_cost 是访问多次s2 表后累加起来的值，主要关注里边的prefix_cost 的值代表的是整个连接查询预计的成本，也就是单次查询s1 表和多次查询s2 表后的成本的和，也就是：

968.80 + 193.76 + 2034.60 = 3197.16
