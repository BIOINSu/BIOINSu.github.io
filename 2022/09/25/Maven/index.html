Maven

发表于 2022-09-25 更新于 2022-10-06 分类于 工具

Maven是一款用于构建项目的工具，只需要告诉Maven需要哪些Jar 包，它就会帮助用户下载所有的Jar，以提升开发效率。

Maven项目的目录结构

Maven仓库

Maven仓库用来存放Maven这个工具管理的所有Jar包。分为本地仓库 和 中央仓库。

本地仓库：Maven本地的Jar包仓库。 中央仓库： Maven官方提供的远程仓库。

当项目编译时，Maven首先从本地仓库中寻找项目所需的Jar包，若本地仓库没有，再到Maven的中央仓库下载所需Jar包。

Maven并不会每次都从中央仓库下载jar包。一个jar包一旦被下载过，就会被Maven自动缓存在本地目录（用户主目录的.m2目录），所以，除了第一次编译时因为下载需要时间会比较慢，后续过程因为有本地缓存，并不会重复下载相同的jar包。

pom文件

pom文件，即project object model 项目对象模型，包含了项目的各种配置信息。

一个pom文件中各个节点的配置信息通常包括：

基本配置信息

节点 解释说明
modelVersion 描述当前 pom 文件遵从哪个pom模型版本，maven2和3只能为4.0.0
groupId 组织标识，通常是公司域名
artifactId 项目名称，又或是模块的名称，类似于Java的类名。
version 这是项目的版本。定义一个组件/模块为快照版本，只需要在pom文件中在该模块的版本号后加上-SNAPSHOT即可(注意这里必须是大写，只能是这种格式)。
release版本不允许修改，每次进行release版本修改，发布必须提升版本号。而snapshot一般是开发过程中的迭代版本，snapshot更新后，引用的项目可以不修改版本号自动下载构建。
packaging 项目打包方式，有以下值：pom, jar, maven-plugin, ejb, war, ear, rar, par。
父模块中的pom.xml文件使用的packaging配置一定为pom，它只作项目的子模块的整合，在maven install时不会生成jar/war压缩包。
jar是最为常见的打包方式，当pom文件中没有设置packaging参数时，默认使用jar方式打包。这种打包方式意味着在maven build时会将这个项目中的所有java文件都进行编译形成.class文件，且按照原来的java文件层级结构放置，最终压缩为一个jar文件。

依赖配置

dependencies

这个标签声明了当前项目或者模块需要依赖到哪些jar包。

只有以-SNAPSHOT结尾的版本号会被Maven视为开发版本，开发版本每次都会重复下载。Maven 在本地是进行缓存的，如果使用的是 release 版本的话，一旦包被下载到本地了，Maven 是不会重新下载的，如果希望重新下载的话，你需要手动删除本地中的缓存。如果你使用的是 SNAPSHOT 版本的话，每次在构建的时候都会从定义的新仓库中下载新的版本。

例如项目小组 A 有个包的版本为 0.0.1-SNAPSHOT ，项目小组 B 中有个包是依赖于这个的，但是项目小组 B 发现这个包中有问题，需要进行修改。小组 B 将会通知小组 A 进行修改，小组 A 修改完成后将会 Push 到仓库中，尽管版本号没有变，那么项目小组 B 在编译的时候，也会使用新推送的 0.0.1-SNAPSHOT 版本。

标签声明了当前项目所依赖的这个jar包的作用范围，可选值如下:

节点 解释说明
compile 默认值，表示被依赖的项目需要参与当前项目的编译，当然后续的测试，运行周期也参与其中，是一个比较强的依赖。打包的时候通常需要包含进去。
provided 表明当前项目打包的时候可以不用将这个依赖的jar包打进去，别的设施(容器、JDK等等)会提供。事实上该依赖可以参与编译，测试，运行等周期。相当于compile，但是在打包阶段做了exclude的动作。
runtime 表示被依赖项目无需参与项目的编译，不过后期的测试和运行周期需要其参与。
test 依赖在一般的编译和运行时都不需要，它们只有在测试编译和测试运行阶段可用。

compile 、provided 、runtime 的区别

通过 maven 引入的jar包，里面的类，都是已经编译好的字节码文件。

  • 通过 compile 和 provided 引入的 jar 包，里面的类在项目中可以直接 import 引用使用，编译没问题，

  • 通过 runtime 引入的 jar 包中的类，只在运行时生效，所以项目中不能直接 import 使用，只能通过反射等方式使用。

  • 通过 compile 和 runtime 引入的 jar 包，会一起打包到自己项目包里，而 provided 引入的 jar 包则不会。

parent

这个标签声明了当前项目或者模块的父项目。

节点 解释说明
groupId 父项目的组织标识符
artifactId 父项目的名称，也是它的唯一标识符
relativePath Maven首先在当前项目中找父项目的pom，然后在文件系统的这个位置（relativePath），然后在本地仓库，再在远程仓库找。
version 父项目的版本。

在子模块中声明了parent标签后，可以直接继承父模块dependency标签内声明的依赖（dependencyManagement的依赖还需要显式声明）。父模块坐标中的packaging必须是pom类型，否则子模块就不会继承。

dependencyManagement

配置写法同dependencies。

在父项目dependencies中定义的所有依赖，子项目会直接继承，可以直接使用。

在父项目dependencyManagement中定义的所有依赖，子项目不会直接继承，需要显示地声明，但是可以不用指定版本号。这样的好处是，父项目统一了版本，而且子项目可以在需要的时候才引用对应的依赖。此时子项目也可以自己定义版本号，这样在子项目中使用的就是另一个版本的jar包。

modules

当该maven项目被组织成多个模块时，这个标签用于指定当前项目所包含的所有模块。之后对这个项目进行的maven操作，会让所有子模块也进行相同操作。

基本概念

jar包

jar包即 Java Archive File，它将一系列文件合并为一个文件，类似于ZIP文件。jar包主要是对java项目编译生成的.class文件以及其他静态资源文件如.html、.css以及.js等项目所需的一切进行打包，而.class文件是平台无关的，即就意味着jar包是跨平台的。

jar 与 zip 高度相似，唯一的区别就是在 jar 文件的内容中，包含了一个 META-INF/MANIFEST.MF 文件，该文件是在生成 jar 文件的时候自动创建的，它包含了该Jar包的版本、创建人和各个类的路径Class-Path等信息，如果是可执行Jar包，会包含Main-Class属性，表明Main方法入口。

当开发了一个程序以后，程序中有很多的类，如果需要提供给别人使用，发给对方一大堆源文件是非常不好的，因此通常需要把这些类以及相关的资源文件打包成一个 jar 包，把这个 jar 包提供给别人使用，同时提供给使用者清晰的文档。这样他人在拿到我们提供的jar之后，就能方便地进行调用，具体如何调用后面会进行讲解。
